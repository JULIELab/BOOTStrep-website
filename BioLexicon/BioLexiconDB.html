BioLexicon

BioLexicon DB

The BioLexicon has been implemented in a MySQL DB consisting of two modules: the actual MySQL database, i.e. the container of terms and term related information, and a java software component for the automatic population of the database. The database is structured into three logically distinct but strongly interconnected layers. The neat separation between the tables containing the data (the BioLexicon proper) and “operational” tables allows for the optimization of the uploading of data into the BioLexicon DB, and ensures an easy extendibility both of the database and of the uploading procedures. Face to the need to add a new table (i.e. a new lexical object), or a new attribute to an existing table, it is sufficient to add only the properties of the new tables or attributes and to modify accordingly only the relevant portion of the DB layer to handle the novelties.

BioLexicon distribution for research use by academia, for research use by commercial organisations and for commercial use, is undertaken on behalf of the developers by the European Language Resources Association Distribution Agency (ELDA)

http://www.elda.org/

The BioLexicon Web Interface

The BioLexicon DB comes equipped together with a dedicated user-friendly web interface which presents the user a set of possible typical queries. The interface allows visualization of both biological and linguistic information of terms encoded in the lexical resource. It also allows the user to simply navigate through terms, their relations and their lexical forms, using a hyperlink-based structure. Different search criteria let the user browse terms using a word matching search engine, with the possibility to choose different data sources and/or semantic/lexical types. A search by identifier is also available. A help menu displays documentation or any other information available about the properties involved in the description of a term. Two different levels of description are proposed for each term and property: a biological description and a lexical one. A JavaDoc is available for developers interested in querying directly the database with the Java APIs. Moreover, a FAQ section describes all the common troubleshooting that may arise and their solutions.

http://wiki.ilc.cnr.it/BootStrep/searchPanel.action

XIF and Automatic Up-loading procedures

External to the DB but fundamental for its automatic population is an XML Interchange Format (XIF) specifically tailored on the BioLexicon structure. This interchange format has been designed in the Bootstrep project with the main purpose of automatically populating the BioLexicon with data provided by domain experts. The XML Interchange Format DTD is to be considered a simplified version of the BioLexicon DTD, which on the one hand accommodates the needs of data providers and, on the other, facilitate the automatic uploading of the DB. By means of the XIF, standardization of the data extracted from the different terminological resources and from texts (by automatic NLP applications) is allowed. This way, any system/group wishing to feed new data into the BioLexicon would only need to encode this in an XML file according to the XIF DTD.

http://www.ilc.cnr.it/viewpage.php/sez=ricerca/id=860/vers=ing

The loading software interprets the XIF and applies the new instructions automatically inserting the new table or attribute. This property of the XIF together with the neat separation of the three layers of the DB mentioned above allows any agent (human or machine) to easily populate, update and create new objects and attributes.

http://www.ilc.cnr.it/viewpage.php/sez=ricerca/id=860/vers=ita

BioLexicon WebServices

This web service extracts the linguistic layers information from the BioLexicon database: it is based on SOAP protocol. The web service extracts a LemId for a given entry: the user can query for a LemId using two different APIs: getLemmaFromSourceVal and getLemmaFromWord. The first one extracts all lemma information (LemdId included,) that matches the source value, whereas the latter allows filtering all lemma information by different criteria: POS, Semantic Type, Sources, word (exact match or substring, case sensitive/insensitive). Once a LemId is retrieved, the user is able to obtain different linguistic information (variants, inflected forms, data categories, syntax, semantics and syntax-semantic mapping) simply invoking the service API with a LemID as a parameter. The results of each operation consist of an operation status structure and a list of the results.

http://wiki.ilc.cnr.it/BootStrep/webservices.html
